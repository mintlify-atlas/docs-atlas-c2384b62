## Overview

The **Contractor** role is automatically assigned to users who have a record in the `info_contratista` table. Contractors are external personnel providing services to Universidad Cooperativa de Colombia through third-party companies.

<Info>
The Contractor role is managed automatically through database triggers. When a record is added to `info_contratista`, the system automatically assigns the Contractor role to that user.
</Info>

## Database Schema

The `info_contratista` table stores minimal contractor information needed for access control:

```sql
CREATE TABLE info_contratista (
    id               BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    id_institucional VARCHAR(20)  NOT NULL UNIQUE,
    empresa          VARCHAR(200) NOT NULL
);
```

### Fields

<ParamField path="id" type="BIGINT">
  Auto-generated primary key identifier
</ParamField>

<ParamField path="id_institucional" type="VARCHAR(20)" required>
  Foreign key reference to `usuarios.id_institucional`. This is the institutional ID assigned by UCC to the contractor.
  
  - Must be unique
  - References the main users table
  - Cascading delete: if the user is deleted, the contractor info is also deleted
</ParamField>

<ParamField path="empresa" type="VARCHAR(200)" required>
  Legal name (razón social) of the contracting company.
  
  Examples:
  - "Servicios Generales S.A.S"
  - "Tecnología y Consultoría Ltda"
  - "Mantenimiento Integral Colombia"
  - "Seguridad Privada Nacional"
</ParamField>

## Design Rationale

<Card title="Company Identification" icon="building">
The system stores **only the company name** — sufficient for security personnel to know which contracting company the person represents, without exposing sensitive contract details.
</Card>

The original design included additional fields like:
- Company tax ID (NIT)
- Assigned area (area_asignada)
- Contract start date (fecha_inicio_contrato)
- Contract end date (fecha_fin_contrato)
- Contract value
- Specific contract terms

However, the actual implementation follows a **privacy-by-design principle**. Contract details are:
- Not necessary for basic access control
- Sensitive business information
- Better managed in dedicated contract management systems
- Subject to confidentiality agreements

## Role Assignment

The Contractor role is assigned automatically through triggers defined in `06_triggers_roles.sql`:

1. **When a record is inserted** into `info_contratista`, the trigger assigns the "Contratista" role
2. **When a record is deleted** from `info_contratista`, the trigger revokes the "Contratista" role
3. Contractors typically have **only one role**, though the system supports multiple roles if needed

## Data Management

The `info_contratista` table is updated by the **Contract Management** department:

<Steps>
  <Step title="Contract Award">
    When a service contract is awarded, Contract Management creates records for authorized personnel
  </Step>
  <Step title="Personnel Registration">
    Each contractor receives an institutional ID and is registered with their company affiliation
  </Step>
  <Step title="Role Auto-Assignment">
    Database trigger automatically assigns the Contractor role
  </Step>
  <Step title="Contract Completion">
    When contract ends, Contract Management deletes records, trigger revokes Contractor role
  </Step>
</Steps>

<Warning>
Contractors must be pre-registered by authorized UCC staff. Security personnel cannot register new contractors on the spot.
</Warning>

## Example Queries

### Get all contractors

```sql
SELECT u.id_institucional, u.nombre_completo, c.empresa
FROM usuarios u
JOIN info_contratista c ON u.id_institucional = c.id_institucional
WHERE u.estado_general = 'activo';
```

### Get contractor info for a specific user

```sql
SELECT empresa
FROM info_contratista
WHERE id_institucional = '80123456';
```

### Count contractors by company

```sql
SELECT empresa, COUNT(*) as total_contratistas
FROM info_contratista
GROUP BY empresa
ORDER BY total_contratistas DESC;
```

### List all personnel from a specific company

```sql
SELECT u.nombre_completo, u.id_institucional, u.estado_general
FROM info_contratista c
JOIN usuarios u ON c.id_institucional = u.id_institucional
WHERE c.empresa = 'Servicios Generales S.A.S'
ORDER BY u.nombre_completo;
```

## Access Control Impact

When a contractor attempts entry without their physical ID card:

1. The system displays their **institutional ID** and **full name**
2. The **Contractor** role badge appears
3. The **company name (empresa)** is shown to verify affiliation
4. Security personnel can confirm the contractor's company before granting access

<Note>
Security personnel should verify that the contractor is from a company currently under contract with UCC.
</Note>

## Important Considerations

### Access Limitations

<Warning>
Contractors typically have:
- **Limited access** to specific areas where they provide services
- **Time-restricted** access based on contract terms
- **No administrative privileges** in the system
</Warning>

### Security Verification

When a contractor arrives without their ID card, security should:

<Steps>
  <Step title="Verify Identity">
    Check government-issued ID (cédula) matches the name in the system
  </Step>
  <Step title="Confirm Company">
    Verify the contractor belongs to a company currently providing services to UCC
  </Step>
  <Step title="Check Status">
    Ensure the user is not blocked due to previous incidents
  </Step>
  <Step title="Record Entry">
    Register the contingent entry with appropriate reason (olvido/pérdida/robo)
  </Step>
</Steps>

### Contract Expiration

<Info>
When a service contract ends, Contract Management is responsible for:
- Removing contractor records from the database
- Collecting any UCC-issued ID cards
- Updating building access permissions
</Info>

## Multi-Role Scenarios

Though rare, contractors can hold multiple roles:

```
Example: Juan Martínez (ID: 80345678)
├─ 🏗️ CONTRACTOR
│  └─ Empresa: Tecnología y Consultoría Ltda
│
└─ 🎓 STUDENT
   └─ Programa: Especialización en Gerencia de Proyectos
```

This allows external personnel who are also enrolled in academic programs to be properly identified in both capacities.
