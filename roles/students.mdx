## Overview

The **Student** role is automatically assigned to users who have a record in the `info_estudiante` table. Students are individuals enrolled in academic programs at Universidad Cooperativa de Colombia.

<Info>
The Student role is managed automatically through database triggers. When a record is added to `info_estudiante`, the system automatically assigns the Student role to that user.
</Info>

## Database Schema

The `info_estudiante` table stores minimal academic information needed for access control:

```sql
CREATE TABLE info_estudiante (
    id               BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    id_institucional VARCHAR(20)  NOT NULL UNIQUE,
    programa         VARCHAR(150) NOT NULL
);
```

### Fields

<ParamField path="id" type="BIGINT">
  Auto-generated primary key identifier
</ParamField>

<ParamField path="id_institucional" type="VARCHAR(20)" required>
  Foreign key reference to `usuarios.id_institucional`. This is the institutional ID assigned by UCC (e.g., "80123456").
  
  - Must be unique
  - References the main users table
  - Cascading delete: if the user is deleted, the student info is also deleted
</ParamField>

<ParamField path="programa" type="VARCHAR(150)" required>
  Name of the academic program the student is enrolled in.
  
  Examples:
  - "Ingeniería de Sistemas"
  - "Derecho"
  - "Administración de Empresas"
  - "Medicina"
</ParamField>

## Design Rationale

<Card title="Minimal Data Approach" icon="shield-check">
The system stores **only the academic program** — sufficient information for security personnel to identify what the person studies, without exposing unnecessary sensitive academic data.
</Card>

The original design included additional fields like:
- Faculty (facultad)
- Campus/sede
- Semester number
- Academic status

However, the actual implementation follows a **privacy-by-design principle** by storing only what's necessary for access control purposes. Additional academic details represent an unnecessary risk of data exposure.

## Role Assignment

The Student role is assigned automatically through triggers defined in `06_triggers_roles.sql`:

1. **When a record is inserted** into `info_estudiante`, the trigger assigns the "Estudiante" role
2. **When a record is deleted** from `info_estudiante`, the trigger revokes the "Estudiante" role
3. Users can have **multiple roles simultaneously** (e.g., a student who also works as an employee)

## Data Loading

The `info_estudiante` table is typically reloaded at the **beginning of each semester** using CSV imports from the academic system.

<Steps>
  <Step title="Export from Academic System">
    Extract current student enrollment data with institutional IDs and programs
  </Step>
  <Step title="Prepare CSV File">
    Format the data with columns: `id_institucional`, `programa`
  </Step>
  <Step title="Clear Previous Data">
    Remove outdated student records from the previous semester
  </Step>
  <Step title="Import New Data">
    Load the CSV file to update student information for the new semester
  </Step>
</Steps>

## Example Queries

### Get all students

```sql
SELECT u.id_institucional, u.nombre_completo, i.programa
FROM usuarios u
JOIN info_estudiante i ON u.id_institucional = i.id_institucional
WHERE u.estado_general = 'activo';
```

### Get student info for a specific user

```sql
SELECT programa
FROM info_estudiante
WHERE id_institucional = '80123456';
```

### Count students by program

```sql
SELECT programa, COUNT(*) as total_estudiantes
FROM info_estudiante
GROUP BY programa
ORDER BY total_estudiantes DESC;
```

## Access Control Impact

When a student attempts entry without their physical ID card:

1. The system displays their **institutional ID** and **full name**
2. The **Student** role badge appears
3. The **academic program** is shown to help security verify identity
4. Security personnel can authorize contingent entry after verification

<Warning>
This system is **only used when the student does NOT have their physical ID card**. Normal entries with valid ID cards are not recorded in this system.
</Warning>
