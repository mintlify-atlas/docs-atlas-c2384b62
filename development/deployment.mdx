## Overview

This guide covers building the application for production and deploying it to various hosting platforms. The UCC Control de Acceso system is a static single-page application (SPA) that can be deployed to any static hosting service.

## Build Process

### Production Build

The Vite build process optimizes your application for production:

<Steps>
  <Step title="Clean previous builds">
    ```bash
    rm -rf dist/
    ```
  </Step>

  <Step title="Run production build">
    <CodeGroup>
      ```bash npm
      npm run build
      ```

      ```bash yarn
      yarn build
      ```

      ```bash pnpm
      pnpm build
      ```
    </CodeGroup>

    This command:
    - Bundles all JavaScript and CSS files
    - Minifies and optimizes assets
    - Generates source maps (configured in `vite.config.js`)
    - Outputs to `dist/` directory
  </Step>

  <Step title="Verify build output">
    ```bash
    ls -lh dist/

    # Expected output:
    # index.html
    # assets/
    #   ├── index-[hash].js
    #   ├── index-[hash].css
    #   └── [images/fonts]
    ```

    <Check>
      The `dist/` folder should contain an `index.html` file and an `assets/` directory with hashed filenames for cache busting.
    </Check>
  </Step>
</Steps>

### Build Configuration

The build is configured in `vite.config.js`:

```javascript vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  server: {
    port: 3000,
    open: true
  },
  build: {
    outDir: 'dist',           // Output directory
    sourcemap: true,          // Generate source maps for debugging
    minify: 'esbuild',        // Fast minification
    target: 'es2015',         // Browser compatibility target
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['react', 'react-dom', 'react-router-dom'],
          supabase: ['@supabase/supabase-js'],
        }
      }
    }
  }
});
```

### Preview Production Build Locally

Before deploying, test the production build locally:

<CodeGroup>
  ```bash npm
  npm run preview
  ```

  ```bash yarn
  yarn preview
  ```

  ```bash pnpm
  pnpm preview
  ```
</CodeGroup>

This starts a local server serving the `dist/` directory at `http://localhost:4173`.

<Note>
  The preview command simulates a production environment. Use it to catch any build-related issues before deploying.
</Note>

## Environment Variables

### Production Environment Setup

Create production environment variables:

<Steps>
  <Step title="Configure production .env">
    Create a `.env.production` file:

    ```bash .env.production
    # Production Supabase Configuration
    VITE_SUPABASE_URL=https://your-production-project.supabase.co
    VITE_SUPABASE_ANON_KEY=your-production-anon-key
    
    # API URL (if using external API)
    VITE_API_URL=https://api.your-domain.com
    
    # Environment identifier
    VITE_ENVIRONMENT=production
    ```
  </Step>

  <Step title="Build with production variables">
    Vite automatically loads `.env.production` when building:

    ```bash
    npm run build
    ```

    Or specify environment explicitly:

    ```bash
    NODE_ENV=production npm run build
    ```
  </Step>
</Steps>

<Warning>
  **Never commit `.env.production` to version control.** Add it to `.gitignore`. Use your hosting platform's environment variable settings instead.
</Warning>

## Deployment Options

### Option 1: Vercel (Recommended)

Vercel offers excellent Vite support with zero configuration:

<Steps>
  <Step title="Install Vercel CLI">
    ```bash
    npm install -g vercel
    ```
  </Step>

  <Step title="Login to Vercel">
    ```bash
    vercel login
    ```
  </Step>

  <Step title="Deploy to production">
    ```bash
    vercel --prod
    ```

    Follow the prompts to configure your project.
  </Step>

  <Step title="Set environment variables">
    In Vercel dashboard:
    1. Go to **Project Settings** > **Environment Variables**
    2. Add your `VITE_*` variables
    3. Redeploy: `vercel --prod`
  </Step>
</Steps>

**Vercel Configuration** (optional `vercel.json`):

```json vercel.json
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "framework": "vite",
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/index.html"
    }
  ]
}
```

### Option 2: Netlify

Netlify provides continuous deployment from Git:

<Steps>
  <Step title="Create netlify.toml">
    ```toml netlify.toml
    [build]
      command = "npm run build"
      publish = "dist"

    [[redirects]]
      from = "/*"
      to = "/index.html"
      status = 200

    [build.environment]
      NODE_VERSION = "18"
    ```
  </Step>

  <Step title="Deploy via Netlify CLI">
    ```bash
    # Install Netlify CLI
    npm install -g netlify-cli

    # Login
    netlify login

    # Deploy
    netlify deploy --prod
    ```
  </Step>

  <Step title="Or connect Git repository">
    1. Push code to GitHub/GitLab
    2. Import repository in Netlify dashboard
    3. Configure build settings (auto-detected)
    4. Add environment variables in **Site settings** > **Environment variables**
  </Step>
</Steps>

### Option 3: GitHub Pages

Deploy as a static site on GitHub Pages:

<Steps>
  <Step title="Update vite.config.js for base path">
    ```javascript vite.config.js
    export default defineConfig({
      base: '/ucc-control-acceso/',  // Replace with your repo name
      // ... rest of config
    });
    ```
  </Step>

  <Step title="Install gh-pages package">
    ```bash
    npm install --save-dev gh-pages
    ```
  </Step>

  <Step title="Add deploy script to package.json">
    ```json package.json
    {
      "scripts": {
        "dev": "vite",
        "build": "vite build",
        "preview": "vite preview",
        "deploy": "npm run build && gh-pages -d dist"
      }
    }
    ```
  </Step>

  <Step title="Deploy">
    ```bash
    npm run deploy
    ```

    Your app will be available at: `https://<username>.github.io/ucc-control-acceso/`
  </Step>
</Steps>

<Warning>
  GitHub Pages doesn't support environment variables. You'll need to hardcode production Supabase credentials or use a different approach.
</Warning>

### Option 4: Self-Hosted (Docker)

Deploy using Docker for full control:

<Steps>
  <Step title="Create Dockerfile">
    ```dockerfile Dockerfile
    # Build stage
    FROM node:18-alpine AS builder
    WORKDIR /app
    COPY package*.json ./
    RUN npm ci
    COPY . .
    RUN npm run build

    # Production stage
    FROM nginx:alpine
    COPY --from=builder /app/dist /usr/share/nginx/html
    COPY nginx.conf /etc/nginx/conf.d/default.conf
    EXPOSE 80
    CMD ["nginx", "-g", "daemon off;"]
    ```
  </Step>

  <Step title="Create nginx.conf">
    ```nginx nginx.conf
    server {
        listen 80;
        server_name localhost;
        root /usr/share/nginx/html;
        index index.html;

        location / {
            try_files $uri $uri/ /index.html;
        }

        # Cache static assets
        location /assets {
            expires 1y;
            add_header Cache-Control "public, immutable";
        }
    }
    ```
  </Step>

  <Step title="Build and run Docker container">
    ```bash
    # Build image
    docker build -t ucc-control-acceso .

    # Run container
    docker run -d -p 80:80 --name ucc-app ucc-control-acceso
    ```

    Access at `http://localhost`
  </Step>

  <Step title="Deploy with Docker Compose">
    ```yaml docker-compose.yml
    version: '3.8'
    services:
      web:
        build: .
        ports:
          - "80:80"
        environment:
          - NODE_ENV=production
        restart: unless-stopped
    ```

    Deploy:
    ```bash
    docker-compose up -d
    ```
  </Step>
</Steps>

### Option 5: AWS S3 + CloudFront

Deploy to AWS for enterprise-grade hosting:

<Steps>
  <Step title="Build the application">
    ```bash
    npm run build
    ```
  </Step>

  <Step title="Create S3 bucket and enable static hosting">
    ```bash
    aws s3 mb s3://ucc-control-acceso
    aws s3 website s3://ucc-control-acceso --index-document index.html
    ```
  </Step>

  <Step title="Upload build files">
    ```bash
    aws s3 sync dist/ s3://ucc-control-acceso --delete
    ```
  </Step>

  <Step title="Configure CloudFront distribution">
    Create a CloudFront distribution pointing to your S3 bucket for CDN and HTTPS support.
  </Step>
</Steps>

## Post-Deployment Checklist

After deploying to production, verify:

<Steps>
  <Step title="✅ Test core functionality">
    - Login with institutional ID works
    - User data loads correctly
    - Admin panel accessible (if applicable)
    - Failure registration works
    - Reports generate and export
  </Step>

  <Step title="✅ Check Supabase connection">
    - Verify environment variables are set
    - Test database queries return data
    - Confirm RLS policies allow necessary access
  </Step>

  <Step title="✅ Verify performance">
    - Run Lighthouse audit (aim for 90+ score)
    - Check bundle size (use `npm run build -- --analyze`)
    - Test on mobile devices
  </Step>

  <Step title="✅ Security review">
    - HTTPS enabled
    - No sensitive data in client-side code
    - Environment variables properly configured
    - CORS settings correct in Supabase
  </Step>

  <Step title="✅ Browser compatibility">
    Test on:
    - Chrome/Edge (latest)
    - Firefox (latest)
    - Safari (latest)
    - Mobile browsers (iOS Safari, Chrome Android)
  </Step>
</Steps>

## Continuous Deployment (CI/CD)

### GitHub Actions Workflow

Automate deployment with GitHub Actions:

```yaml .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Build
        env:
          VITE_SUPABASE_URL: ${{ secrets.VITE_SUPABASE_URL }}
          VITE_SUPABASE_ANON_KEY: ${{ secrets.VITE_SUPABASE_ANON_KEY }}
        run: npm run build
      
      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
```

**Setup:**
1. Add secrets in GitHub: **Settings** > **Secrets and variables** > **Actions**
2. Add: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VERCEL_TOKEN`, etc.
3. Push to `main` branch triggers automatic deployment

## Performance Optimization

### Code Splitting

Split vendor code for better caching:

```javascript vite.config.js
export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          'react-vendor': ['react', 'react-dom', 'react-router-dom'],
          'ui-vendor': ['recharts'],
          'supabase-vendor': ['@supabase/supabase-js'],
        }
      }
    }
  }
});
```

### Asset Optimization

```javascript vite.config.js
export default defineConfig({
  build: {
    // Optimize images
    assetsInlineLimit: 4096,  // Inline assets < 4kb
    
    // Chunk size warnings
    chunkSizeWarningLimit: 1000,
  }
});
```

### Compression

Enable gzip/brotli compression in nginx:

```nginx nginx.conf
http {
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml;
    gzip_min_length 1000;
    
    brotli on;
    brotli_types text/plain text/css application/json application/javascript;
}
```

## Rollback Strategy

If deployment fails or issues arise:

<Steps>
  <Step title="Vercel/Netlify rollback">
    Use the platform's built-in rollback:
    ```bash
    vercel rollback  # Vercel
    netlify rollback # Netlify
    ```
  </Step>

  <Step title="Git-based rollback">
    ```bash
    git revert HEAD
    git push origin main  # Triggers new deployment
    ```
  </Step>

  <Step title="Manual rollback">
    Redeploy a previous build:
    ```bash
    git checkout <previous-commit>
    npm run build
    # Deploy dist/ manually
    ```
  </Step>
</Steps>

## Troubleshooting

### Build fails with "out of memory"

Increase Node.js memory limit:

```bash
NODE_OPTIONS=--max-old-space-size=4096 npm run build
```

### 404 errors on page refresh (SPA routing)

Configure your hosting to redirect all routes to `index.html`:

- **Netlify**: Add `_redirects` file with `/* /index.html 200`
- **Nginx**: Use `try_files $uri /index.html`
- **Apache**: Use `.htaccess` with rewrite rules

### Environment variables not working

- Verify variable names start with `VITE_`
- Restart dev server after changing `.env`
- For production, set variables in hosting platform dashboard

### Supabase connection errors in production

- Check CORS settings in Supabase dashboard
- Verify production URL is whitelisted
- Confirm environment variables are set correctly
