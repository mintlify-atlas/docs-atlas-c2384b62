## Overview

The UCC Access Control System tracks **TIC card failures** — incidents when users cannot present their physical Tarjeta de Identificación Cooperativista (TIC). The system records two types of failures:

- **Olvido** (Forgot): User forgot to bring their TIC card
- **Pérdida** (Lost): User lost their TIC card

<Note>
All failures are tracked in the `fallas` table and contribute to the user's `total_fallas` counter, which determines if automatic blocking occurs.
</Note>

## Failure Reporting Flow

When a user logs in without their TIC card:

<Steps>
  <Step title="Authentication">
    User enters their institutional ID and is authenticated.
  </Step>
  <Step title="Profile display">
    System displays user information, current roles, and accumulated failures.
  </Step>
  <Step title="Failure type selection">
    User is presented with a modal to select the failure reason:
    - **Olvidé mi TIC** (I forgot my TIC)
    - **Perdí mi TIC** (I lost my TIC)
  </Step>
  <Step title="Failure registration">
    System records the failure in the database and updates the user's failure counter.
  </Step>
  <Step title="Access decision">
    System evaluates the failure count and grants or denies access accordingly.
  </Step>
</Steps>

## Database Implementation

### Failures Table

The `fallas` table stores all failure incidents:

```sql database/supabase/07_fallas.sql
CREATE TABLE fallas (
    id               BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    id_institucional VARCHAR(20)  NOT NULL
                     REFERENCES usuarios(id_institucional) ON DELETE CASCADE,
    fecha_hora       TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    motivo           VARCHAR(10)  NOT NULL
                     CHECK (motivo IN ('olvido', 'perdida'))
);
```

| Column | Type | Description |
|--------|------|-------------|
| `id` | BIGINT | Unique failure record ID |
| `id_institucional` | VARCHAR(20) | User who reported the failure |
| `fecha_hora` | TIMESTAMPTZ | Timestamp when failure was recorded |
| `motivo` | VARCHAR(10) | Failure type: `olvido` or `perdida` |

### Automatic Counter Update

When a failure is recorded, a **database trigger** automatically updates the user's failure counter:

```sql database/supabase/07_fallas.sql
CREATE OR REPLACE FUNCTION fn_actualizar_fallas()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
    v_id_inst   VARCHAR(20);
    v_total     INT;
BEGIN
    -- Get the affected user's ID
    v_id_inst := CASE WHEN TG_OP = 'DELETE' THEN OLD.id_institucional
                                             ELSE NEW.id_institucional END;

    -- Count current failures
    SELECT COUNT(*) INTO v_total
    FROM fallas
    WHERE id_institucional = v_id_inst;

    -- Update counter and auto-block if 4+ failures
    UPDATE usuarios
    SET total_fallas = v_total,
        acceso       = CASE WHEN v_total >= 4 THEN 'bloqueado' ELSE acceso END
    WHERE id_institucional = v_id_inst;

    RETURN NULL;
END;
$$;

-- Trigger after INSERT
CREATE TRIGGER trg_fallas_insert
    AFTER INSERT ON fallas
    FOR EACH ROW
    EXECUTE FUNCTION fn_actualizar_fallas();

-- Trigger after DELETE (for corrections)
CREATE TRIGGER trg_fallas_delete
    AFTER DELETE ON fallas
    FOR EACH ROW
    EXECUTE FUNCTION fn_actualizar_fallas();
```

<Info>
The trigger ensures that `total_fallas` is always accurate and automatically blocks users when they reach 4 failures.
</Info>

## Querying Failure Data

### Get failures for a specific user

```sql
SELECT 
    id,
    fecha_hora,
    motivo
FROM fallas
WHERE id_institucional = '80123456'
ORDER BY fecha_hora DESC;
```

### Count failures by type this month

```sql
SELECT 
    motivo,
    COUNT(*) as total
FROM fallas
WHERE fecha_hora >= DATE_TRUNC('month', CURRENT_DATE)
GROUP BY motivo;
```

### Find users at high risk (3 failures)

```sql
SELECT 
    u.id_institucional,
    u.nombre_completo,
    u.total_fallas
FROM usuarios u
WHERE u.total_fallas = 3
  AND u.acceso = 'activo'
ORDER BY u.nombre_completo;
```

## Failure Correction

If a failure was recorded by mistake, administrators can delete it from the database:

```sql
-- Delete a specific failure
DELETE FROM fallas WHERE id = 123;

-- The trigger automatically recalculates total_fallas
```

<Warning>
Deleting failures should be done carefully and only by authorized administrators. All administrative actions should be logged in the audit system.
</Warning>

## Important Business Rules

1. **Both failure types count equally** toward the 4-failure limit
2. **Failures never expire** — the counter persists until:
   - An administrator manually unblocks the user (resets to 0)
   - A new semester begins and data is refreshed
3. **Automatic blocking** occurs at exactly 4 failures (see [Automatic Blocking](/features/automatic-blocking))
4. **No retroactive changes** — once a failure is recorded with a timestamp, it cannot be modified (only deleted)
